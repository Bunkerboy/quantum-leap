Wat er precies misloopt

Elke WordPress-update begint met het aanmaken van een verborgen bestand .maintenance in de rootmap. Zo lang dat bestand aanwezig is, toont WordPress aan bezoekers: "Tijdelijk niet beschikbaar wegens gepland onderhoud." Normaal verwijdert WordPress dit bestand zelf na de update. Wordt de update onderbroken — door een time-out, een trage server of een verbindingsprobleem — dan blijft het bestand staan en zit uw site permanent in onderhoudsmodus.

Een tweede oorzaak is een plugin-conflict: de nieuwe WordPress-versie introduceert een API-wijziging die een oudere plugin niet ondersteunt, wat resulteert in een fatale PHP-fout en een wit scherm.

1

Maak eerst een back-up via uw hostingpanel

Voordat u ook maar iets aanraakt: maak een back-up van de huidige staat via uw hostingpanel (cPanel → Back-up, of Jetpack/UpdraftPlus als die nog bereikbaar zijn). Zo heeft u altijd een terugkeeroptie als er iets misgaat tijdens het herstel. De meeste hostingproviders bewaren ook automatische snapshots — vraag ernaar als u ze niet vindt.

2

Verwijder het .maintenance-bestand via FTP

Verbind via FTP of SFTP met uw server (gebruik FileZilla of de bestandsbeheerder in cPanel). Navigeer naar de rootmap van uw WordPress-installatie — dit is doorgaans public_html/. Zoek het bestand .maintenance (verborgen bestanden tonen: in FileZilla via Server → Verborgen bestanden tonen). Verwijder het bestand.

Herlaad uw site. Verschijnt de site normaal? Update voltooid. Ziet u nog een fout of wit scherm? Ga door naar stap 3.

3

Deactiveer alle plugins via FTP

Als u het dashboard niet kunt bereiken, deactiveer dan alle plugins via FTP: navigeer naar wp-content/plugins/ en hernoem de map plugins naar plugins_deactivated. WordPress kan de plugins dan niet meer laden en omzeilt het conflict.

Herlaad uw site en dashboard. Werkt alles? Hernoem de map terug naar plugins. De plugins zijn nu aanwezig maar gedeactiveerd in WordPress.

4

Herhaal de update en activeer plugins één voor één

Ga naar Dashboard → Updates en voer de mislukte update opnieuw uit. Activeer daarna uw plugins één voor één via Plugins → Geïnstalleerde plugins. Na elke activatie: herlaad uw site. Gaat de site opnieuw kapot na een specifieke plugin? Die plugin is de boosdoener — zoek een update of alternatief.

5

Verhoog PHP memory limit als de fout aanhoudt

Sommige updates mislukken omdat de PHP memory limit op de server te laag staat. Voeg deze regel toe aan uw wp-config.php (boven de regel /* That's all */):

define('WP_MEMORY_LIMIT', '256M');

Of voeg toe aan .htaccess: php_value memory_limit 256M. Probeer de update daarna opnieuw.

⚠️

Let op: Klik nooit op "Bijwerken" en sluit dan het browservenster. Wacht altijd tot de update volledig afgerond is. Op trage verbindingen of gedeelde hosting kan een update enkele minuten duren.

Herhaling voorkomen: updaten met een veilige workflow

  • Maak altijd een back-up vóór elke update — UpdraftPlus (gratis) doet dit automatisch voor elke update.
  • Update in een staging-omgeving — test updates op een kopie van uw site alvorens op productie te updaten.
  • Update plugins vóór de WordPress-core — zo verkleint u de kans op conflicten.
  • Vermijd updates op drukke momenten — niet vlak voor een presentatie of campagne.

Wanneer schakelt u een professional in?

  • U heeft geen FTP-toegang en het hostingpanel is niet beschikbaar.
  • De site is na het herstellen van plugins nog steeds kapot — een dieperliggend conflict of corrupte bestanden.
  • De update heeft data beschadigd of verwijderd — databaseherstel is dan nodig.
  • U wilt toekomstige updates automatisch en veilig laten verlopen met staging en monitoring.
